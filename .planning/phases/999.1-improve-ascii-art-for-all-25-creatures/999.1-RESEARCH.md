## Summary

This phase is primarily a **content creation** task (26 new ASCII art designs) with two **code changes** required to support the decisions. The content work dominates the timeline.

Two render files currently use `art.append(line, style=template.primary_color)` — treating each art line as plain text and applying a uniform color. This means Rich markup tags embedded in art strings (D-05) would render as literal characters like `[bold red]` rather than styled output. Renderer changes in `render/creatures.py` and `render/battle.py` are mandatory before or alongside art creation.

Additionally, the `CreatureTemplate._validate_ascii_art` model validator uses `len(line)` to enforce the 40-char limit. If art lines contain Rich markup tags, the validator counts tag characters (e.g., `[bold red]` = 10 chars) against the visual width limit. The validator must be updated to strip markup before measuring, or art authors must manually account for invisible markup characters.

The evolution before/after display (`render_evolution_before_after`) currently renders old art above new art sequentially. D-07 requires **side-by-side** display — this requires adding a Rich `Table` or `Columns` layout to `render/evolution.py`.

**Primary recommendation:** Fix the renderer and validator first (Wave 0/1), then create art in rarity-tier batches, then implement side-by-side evolution display.

## Architecture Patterns

### Where Art Is Stored

[VERIFIED: codebase read]

Art lives exclusively in the `ascii_art` field of creature JSON files at `src/devmon/data/creatures/*.json`. The field type is `list[str]` — each string is one line of terminal art. No separate art files or art modules exist. All 26 creatures follow this pattern.

### How Art Is Currently Rendered

[VERIFIED: render/creatures.py lines 57-61, render/battle.py lines 117-121]

Both render locations use the same pattern:

```python
# render/creatures.py and render/battle.py — CURRENT (uniform style, no markup)
art = Text()
for i, line in enumerate(template.ascii_art):
    if i > 0:
        art.append("\n")
    art.append(line, style=template.primary_color)
```

This applies `template.primary_color` as a uniform Rich style to the entire line. Any Rich markup tags in the string (e.g., `[bold red]`) will render as literal characters, not as style markup.

### Required Renderer Change (D-05)

[VERIFIED: uv run python test]

To support per-character markup in art strings, the renderer must switch from `Text.append(line, style=...)` to `Text.from_markup(line)`:

```python
# render/creatures.py and render/battle.py — REQUIRED CHANGE
art = Text()
for i, line in enumerate(template.ascii_art):
    if i > 0:
        art.append("\n")
    art.append_text(Text.from_markup(line))
```

`Text.from_markup()` parses Rich markup tags and creates a styled Text object. Plain lines with no markup tags work identically to `Text.append(line)` — no behavior change for existing art. This change is required in **both** render files.

**Files to change:**
- `src/devmon/render/creatures.py` (lines 58-61)
- `src/devmon/render/battle.py` (lines 118-121)

### Required Validator Change (D-05 + D-04)

[VERIFIED: models/creature.py lines 135-152]

The current art validator measures visual width with `len(line)`, which counts markup tag characters. An art line like `[bold red]~~[/bold red]` (visual width 2) would measure as 20 chars. The validator must strip markup before measuring:

```python
# models/creature.py — REQUIRED CHANGE
import re

@model_validator(mode="after")
def _validate_ascii_art(self) -> "CreatureTemplate":
    """Enforce ASCII art constraints for safe rendering in 80-col terminals."""
    lines = self.ascii_art
    if len(lines) < 3:
        raise ValueError("ASCII art must have at least 3 lines")
    if len(lines) > 20:
        raise ValueError("ASCII art must not exceed 20 lines")
    # Strip Rich markup tags before measuring visual width
    _markup_re = re.compile(r'\[[^\[\]]*\]')
    for i, line in enumerate(lines):
        visual_line = _markup_re.sub('', line)
        if len(visual_line) > 40:
            raise ValueError(
                f"ASCII art line {i + 1} exceeds 40-char limit ({len(visual_line)} visual chars)"
            )
    return self
```

**File to change:** `src/devmon/models/creature.py` (validator only, no model field changes)

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Per-character color in art | Custom ANSI escape sequences | Rich markup + `Text.from_markup()` | Rich handles terminal compat, reset codes, and Windows console |

## Common Pitfalls

### Pitfall 1: Markup Tags Count Toward Line Width in Validator
**What goes wrong:** Art author creates a line like `[bold red]  /\  [/bold red]` (visual width 6). `len(line)` = 28, validator rejects it as too wide.
**Why it happens:** Validator uses `len(line)` before the fix.
**How to avoid:** Fix the validator to strip markup before measuring (see Architecture Patterns above). Do this in Wave 0 before any art is written.
**Warning signs:** `ValidationError: ASCII art line N exceeds 40-char limit` when art looks visually narrow.

### Pitfall 2: Markup in Art Renders as Literal Text
**What goes wrong:** Art contains `[bold red]~~[/bold red]` but it displays as the literal string `[bold red]~~[/bold red]` on screen.
**Why it happens:** Renderer uses `Text.append(line, style=...)` which treats content as plain text.
**How to avoid:** Fix the renderer in both `render/creatures.py` and `render/battle.py` before writing markup-containing art. Test with `uv run devmon battle` after each art update.
**Warning signs:** Visible `[` and `]` characters in terminal output during battle or collection view.

### Pitfall 3: Rich Markup Tags Break Backslash-Heavy Art
**What goes wrong:** Art using `\` characters for creature bodies accidentally creates what looks like escape sequences or malformed markup.
**Why it happens:** `Text.from_markup()` interprets `[` and `]` as markup delimiters. A line like `\[shadow\]` or just `[` in art that isn't closed may raise a `MarkupError`.
**How to avoid:** Escape literal square brackets in art strings as `\[` and `\]`, or avoid them in favor of `(`, `)`, `<`, `>`, `{`, `}` for art construction. Test each JSON file with `uv run python -c "from rich.text import Text; Text.from_markup('[art line here]')"`.
**Warning signs:** `rich.errors.MarkupError` at startup when creature data loads.

### Pitfall 4: Side-by-Side Panels Overflow Narrow Terminals
**What goes wrong:** Two creature panels side-by-side at 20+ chars each overflow a 40-col terminal.
**Why it happens:** Rich Table doesn't know terminal width unless told.
**How to avoid:** The `narrow` flag (terminal width < 40) already exists throughout the codebase. Pass it to `render_evolution_before_after` and fall back to vertical stacking in narrow mode (existing pattern from `render/creatures.py`).
**Warning signs:** Wrapped text or truncated panels in evolution display.

### Pitfall 5: Art Height Mismatch Breaks Evolution Side-by-Side
**What goes wrong:** Old creature art is 5 lines, new evolved form is 14 lines. Side-by-side Table shows mismatched heights and looks broken.
**Why it happens:** Rich Table grid cells align at the top by default.
**How to avoid:** This is acceptable behavior — Rich Table handles height differences naturally (shorter column gets blank space below). No padding needed. Per D-08, Claude determines evolved form sizes — planner should document this is a known visual effect, not a bug.

### Pitfall 6: Art Width `> 20` Visual Columns Breaks Panel Layout
**What goes wrong:** Art drawn at 25-30 chars wide forces Rich Panel to expand, pushing the battle side-by-side layout beyond terminal width.
**Why it happens:** D-04 says "Max 20 cols wide" but the existing validator allows 40 chars. The validator limit (40 chars) is the schema constraint; 20 cols is the aesthetic target from the design decision.
**How to avoid:** In practice, draw art within 20 visual columns. The validator is a safety net for extreme cases, not the target width. Art author should preview in terminal before committing.

## Shading Character Reference

[ASSUMED — based on standard ASCII art practice; no tool verification needed]

Recommended characters for ASCII shading and detail, from dark to light:

| Effect | Characters | Notes |
|--------|-----------|-------|
| Dense fill | `█ ▓ ▒ ░` | Unicode block elements — not safe in all terminals |
| Safe dense | `@`, `#`, `%` | High ASCII density, universally supported |
| Medium fill | `*`, `+`, `=`, `~` | Good for body textures, fur, scales |
| Light/outline | `/`, `\`, `|`, `-`, `_` | Structural lines, limbs, edges |
| Fine detail | `.`, `,`, `'`, `` ` `` | Highlights, small features, dots |
| Eyes | `o`, `O`, `0`, `@`, `*`, `^` | Context-dependent |
| Open space | ` ` (space) | Interior of forms |

**Important:** Unicode block characters (`█ ▓ ▒ ░`) work in most modern terminals but may render as boxes or question marks in some Windows console environments. Rich handles this gracefully in most cases. Stick to safe ASCII if cross-platform fidelity is required; this is an [ASSUMED] concern — actual terminal compat depends on the user's environment.
