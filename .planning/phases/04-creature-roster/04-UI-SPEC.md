## Spacing Scale

Rich panels and terminal layouts use character-cell units, not CSS pixels.
The 8-point pixel scale is mapped to character equivalents below.

| Token | Pixel equiv | Char equiv | Usage |
|-------|-------------|------------|-------|
| xs | 4px | 1 char | Inline padding inside Text blocks |
| sm | 8px | 2 chars | Panel padding (Rich default: 1 char each side) |
| md | 16px | 4 chars | Gap between side-by-side panels in Columns |
| lg | 24px | 6 chars | Blank lines between panel groups |
| xl | 32px | 8 chars | Not used in Phase 4 |
| 2xl | 48px | 12 chars | Not used in Phase 4 |
| 3xl | 64px | 16 chars | Not used in Phase 4 |

Exceptions:
- ASCII art line width: max 40 chars per line (enforced by Pydantic validator).
  Rationale: Rich Panel adds 4-char border overhead (| left + right |); 40-char
  art fits safely inside any Phase 5/6 nested panel layout without overflow.
  Source: RESEARCH.md Open Questions #2.
- Touch targets: not applicable (terminal, keyboard-only).

## Typography

Terminal rendering has no font size â€” all text is monospace at the terminal's
native size. "Size" below maps to Rich semantic styling (bold, dim, etc.).

| Role | Rich Style | Usage |
|------|-----------|-------|
| Display | `bold {rarity_color}` | Creature name in panel title |
| Heading | `bold white` | Section titles (e.g., "Stats", "Abilities") |
| Body | `white` | Stat values, flavor text, evolution chain refs |
| Label | `dim {theme.stat_key}` | Stat key labels ("HP", "ATK", "Type") |
| Muted | `dim white` | Panel subtitle: rarity tier + type (e.g., "Common - Fire") |

Source: Extrapolated from existing status.py patterns (stat_key/stat_value
semantic keys in themes.py). Phase 4 follows the same semantic key convention.

Line height: 1 line per row (terminal default). No configurable line spacing.

## Color

This project uses Rich style strings, not hex values. The 60/30/10 split maps
to semantic layer responsibilities in the terminal.

| Role | Rich Style | Usage |
|------|------------|-------|
| Dominant (60%) | terminal background (no fill) | Panel interior â€” default terminal background |
| Secondary (30%) | `dim` overlays, panel borders in `theme["border"]` | Panel borders, stat label dimming |
| Accent (10%) | rarity color per RARITY_COLORS map (see below) | Panel border, creature name title, rarity badge |
| Destructive | `bold red` | Validation error output only |

Accent reserved for: creature panel border color, creature name in panel title,
rarity badge in panel subtitle. Never used for stat values, flavor text, or
loader diagnostic messages.

Source: RESEARCH.md Pattern 5 (render_creature_panel), themes.py THEMES dict.
Destructive: `bold red` matches established project convention (no existing
destructive style in themes.py â€” using Rich standard red for error output).

## Creature Panel Layout Contract

This section is Phase 4-specific and defines the visual structure for the
canonical `render_creature_panel()` function that Phase 5, 6, and 7 will use.

### Single Creature Panel

```
+--[rarity_color]----[Name]----[rarity_color]--+
|  {ascii_art lines, styled with primary_color} |
|                                               |
|  HP     {base_hp}      Type    {type}         |
|  ATK    {base_attack}  SPD     {base_speed}   |
|  DEF    {base_defense} Capture {capture_rate} |
|                                               |
|  {flavor_text â€” dim white, word-wrapped}      |
+--[dim]{rarity.title()} - {type}[/dim]---------+
```

Panel properties:
- `expand=False` â€” panel width matches content, not terminal width (D-07 variable size)
- `border_style=RARITY_COLORS[template.rarity]`
- `title=f"[{rarity_color}]{template.name}[/{rarity_color}]"`
- `subtitle=f"[dim]{template.rarity.title()} - {template.type}[/dim]"`
- `box=rich.box.ROUNDED` â€” softer look consistent with existing panels in status.py

Stat layout: two-column key/value grid using Rich Text with aligned spacing.
Stat key style: `theme["stat_key"]` (dim cyan in neon, dim white in classic).
Stat value style: `theme["stat_value"]` (white in both themes).
Source: status.py stat rendering pattern.

ASCII art rendering rule:
- Each line: `art.append(line, style=template.primary_color)` â€” NEVER embed Rich markup inside art line strings.
- Art lines stored as `list[str]` in JSON â€” plain ASCII, no markup.
- Max 40 chars per line enforced by `@model_validator(mode="after")` in CreatureTemplate.
Source: RESEARCH.md Pitfall 2, Pattern 5.

## Validation Error Display Contract

When `load_all_creatures()` encounters a bad creature JSON file, the error
output follows this contract (Rich Console to stderr):

| State | Visual Treatment |
|-------|-----------------|
| Single field validation error | `[bold red]Creature data error:[/bold red] {filename}: {field}: {msg}` |
| Multiple errors | List each field error on its own line, indented 2 spaces |
| Load completely blocked | `[bold red]Fatal:[/bold red] Creature data failed validation. Fix errors above and retry.` |

Source: RESEARCH.md Pattern (Pydantic ValidationError handling), D-11 (fail fast).
No interactive recovery â€” validation errors are developer-facing, not player-facing.

## Copywriting Contract

Phase 4 is a data layer â€” no interactive player-facing commands are added.
Copy contracts apply to: loader error messages, the `devmon creatures` debug
display (if implemented as a diagnostic tool), and data embedded in creature
JSON files (flavor text).

| Element | Copy |
|---------|------|
| Loader success (silent) | No output â€” creatures load silently at startup |
| Validation error prefix | "Creature data error: {filename}: {field}: {message}" |
| Fatal load failure | "Fatal: Creature data failed validation. Fix errors above and retry." |
| Invalid DEVMON_HOME override | "Warning: DEVMON_HOME/creatures/ override directory not found â€” using bundled data." |
| Creature not found by ID | "Creature '{id}' not found in registry. Available IDs: {list}" |
| Empty roster (zero files) | "No creature data found. Reinstall devmon or check DEVMON_HOME." |
| Flavor text tone | Humorous/meta, dev-culture references. Example: "This creature only appears after midnight coding sessions. It feeds on caffeine fumes." |
| Flavor text length | 1-2 sentences maximum. Fits in panel without scrolling at 80 columns. |
| Creature names | Title-cased display name (e.g., "EmberFox", "Bugbyte", "Nullhound"). |
| Species line | Flavor species name in sentence case (e.g., "Flame Fox", "Byte Hound"). |

Source: D-12 (mixed naming), D-13 (humor tone), RESEARCH.md Pydantic error output.

Destructive actions in Phase 4: none. This phase adds no commands that modify
or delete save data.

## ASCII Art Production Contract

This section defines the constraints art must satisfy to pass `CreatureTemplate`
Pydantic validation and render correctly.

| Constraint | Value | Enforcement |
|-----------|-------|-------------|
| Max line width | 40 characters | `@model_validator(mode="after")` raises ValueError |
| Min lines | 3 lines | `@model_validator(mode="after")` raises ValueError |
| Max lines | 20 lines | `@model_validator(mode="after")` raises ValueError |
| Allowed characters | Printable ASCII (0x20â€“0x7E) | Pydantic str type + validator |
| No Rich markup in art | Plain ASCII only | Validator checks for `[` chars used as tags |
| Backslashes | Must be double-escaped in JSON (`\\`) | Documented in creature file header comment |
| Encoding | UTF-8 JSON files | `read_text(encoding="utf-8")` in loader |

Rationale for 40-char max: Rich Panel border = 4 chars overhead. 40 + 4 = 44,
well within 80-column terminal (CREA-03). Leaves room for Phase 6 battle screen
which places two creature panels side-by-side.

Source: D-07 (variable size), RESEARCH.md Open Questions #2, Pitfall 2, Pitfall 6.
