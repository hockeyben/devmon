## Spacing Scale

Rich terminal UI does not use px spacing. The equivalent is character-count padding and blank-line separation. The following character-based scale mirrors the 8-point system adapted for terminal columns:

| Token | Characters | Usage |
|-------|-----------|-------|
| xs | 1 char | Inline label/value separation (e.g. "HP  " label gap) |
| sm | 2 chars | Between stat columns in two-column stat rows |
| md | 4 chars | Indent for body content inside panels ("  " prefix) |
| lg | 1 blank line | Between panel sections within a Group |
| xl | 2 blank lines | Between distinct notification panels stacked in sequence |
| separator | "  " + "-" * 38 | Divider between categories in list panels (established pattern) |

Exceptions:
- Evolution before/after panels are stacked with 1 blank line between them (lg), not 2, to keep the transformation visually cohesive.
- Narrow terminal mode (width < 40): All indentation reduced to 0 chars. ASCII art hidden entirely. Stats render as single-column.

> Source: Established in src/devmon/render/creatures.py, battle.py, quests.py (audit of existing patterns).

## Typography

Rich terminal UI uses Rich style strings, not font sizes. The following maps semantic roles to Rich style declarations:

| Role | Rich Style | Usage |
|------|-----------|-------|
| Body | `"white"` weight normal | Default narration text, panel body content |
| Label / Dim | `"dim white"` | Stat keys, secondary info, prompt hints, separators |
| Heading | `"bold white"` | Panel headings, creature names in result screens |
| Accent Heading | `"bold {color}"` | Notification panel titles — color varies by notification type (see Color section) |

Line height: 1 line per text segment. Blank lines inserted explicitly via `Text("")` in Group layouts.

Monospace constraint: All text renders in terminal monospace. No font selection available.

> Source: Established across src/devmon/render/*.py. Style strings are the typography system.

## Surfaces

Phase 10 introduces 4 new render surfaces and 1 system-wide adaptation.

### Surface 1: Evolution Prompt

**Trigger:** End of battle victory flow, after XP/level-up processing, when creature has crossed evolution threshold.
**Location:** `src/devmon/render/evolution.py` (new module)
**Pattern:** console.print (not Rich Live — Live must be exited before interactive prompts, established in Phase 6).

**Layout:**
```
┌─ [bold yellow] {CreatureName} wants to evolve! [/bold yellow] ─────────────────────────────┐
│  {CreatureName} has reached level {N} and can evolve into {EvolvedName}!                    │
│                                                                                              │
│  Allow evolution? [y/n]:                                                                    │
└──────────────────────────────────────────────────────────────────────────────────────────────┘
```

**Rich spec:**
- Panel border_style: `"bold yellow"`
- Panel box: `box.ROUNDED`
- Panel expand: `False`
- Body: "bold white" creature name, "white" description line, "dim white" prompt hint
- Prompt line: plain `input()` call after printing panel (matches existing battle prompt pattern)
- Declined response: print `"[dim white]{CreatureName} held back. Maybe next time.[/dim white]"` — no panel

> Source: D-01, D-02 (CONTEXT.md).

### Surface 2: Evolution Before/After Display

**Trigger:** Immediately after player confirms evolution (y).
**Pattern:** Two `render_creature_panel` calls separated by a single blank line, then a confirmation line.

**Layout (top to bottom):**
1. `render_creature_panel(old_template, ...)` — existing function, rarity border
2. Blank line
3. Directional indicator: `Text("  ↓  Evolving...  ↓", style="bold yellow")`
4. Blank line
5. `render_creature_panel(new_template, ...)` — new evolved form, rarity border
6. Blank line
7. Confirmation line: `Text("  {OldName} evolved into {NewName}!", style="bold yellow")`

> Source: D-07 (CONTEXT.md) — "stacked: old creature art + stats → new creature art + stats."

### Surface 3: Evolution Notification (Deferred)

**Trigger:** Next `devmon` invocation after an evolution occurred (if evolution happened outside interactive session — e.g., from condition-based trigger detected on startup).
**Pattern:** Deferred notification panel, same stack as quest/achievement notifications in main.py.

**Rich spec:**
- Panel title: `"[bold yellow]Evolution![/bold yellow]"`
- Panel border_style: `"bold yellow"`
- Panel box: `box.DOUBLE`
- Panel expand: `True`
- Body line 1: `"{OldName} evolved into {NewName}!"` — bold white
- Body line 2: `"New form: {NewName}"` — dim white (flavor, not required if same session)

### Surface 4: Narrow Terminal Adaptation (System-wide)

**Trigger:** `console.width < 40` detected at render time.
**Applies to:** All render modules.

**Rules:**
- ASCII art: Hidden entirely (not shown). No compact variant.
- HP bars: Width reduced to 10 characters (from default 20).
- Stat rows: Single-column layout (no side-by-side two-column format).
- Panel titles: Truncated to 30 characters with "..." suffix if longer.
- Evolution before/after: Show stats only, no ASCII art.

**Detection pattern:**
```python
from rich.console import Console
console = Console()
narrow = console.width < 40
```

Each render function that displays ASCII art accepts an optional `narrow: bool = False` parameter. When `narrow=True`, art block is skipped.

> Source: D-08 (CONTEXT.md), UI-06 (REQUIREMENTS.md) — "All UI respects terminal width and degrades gracefully in narrow terminals."

## Notification Stack Order (D-10)

When multiple deferred notifications are pending, render in this fixed order:

1. Level-up (cyan) — player progression first
2. Evolution (gold) — creature transformation second
3. Quest complete (magenta) — task reward third
4. Daily bonus (cyan) — bonus reward fourth
5. Achievement unlock (magenta) — long-term milestone last

> Source: D-10 (CONTEXT.md) — "Level-up, then evolution, then quest complete, then achievement."

## Copywriting Contract

| Element | Copy |
|---------|------|
| Evolution prompt title | `{CreatureName} wants to evolve!` |
| Evolution prompt body | `{CreatureName} has reached level {N} and can evolve into {EvolvedName}!` |
| Evolution prompt CTA | `Allow evolution? [y/n]:` |
| Evolution declined | `{CreatureName} held back. Maybe next time.` |
| Evolution re-prompt (on next level-up) | `{CreatureName} still wants to evolve into {EvolvedName}. Allow? [y/n]:` |
| Evolution confirmation line | `{OldName} evolved into {NewName}!` |
| Evolution notification title | `Evolution!` |
| Narrow terminal fallback | No copy change — art simply absent, text unchanged |
| Empty evolution state (no evolution available) | Not displayed — prompt only appears when threshold is met |
| Condition-based evolution pending | Not shown to player — evolves silently when condition is met, then notification fires |
| Error: evolved form template not found | `{CreatureName} tried to evolve but something went wrong. (Template missing.)` — dim white, no panel |

**Creature name rule:** Always use `OwnedCreature.display_name` (nickname if set, else template name) in all evolution copy. Consistent with Phase 7 display_name helper pattern.

> Source: D-01, D-02, D-06, D-07 (CONTEXT.md). display_name pattern established in Phase 7.

## Interaction Contracts

### Evolution Prompt Interaction

- **Input:** Single character `y` or `n` (case-insensitive). Any other input treated as `n`.
- **No confirmation loop:** Single input only — no re-prompt on invalid input. Invalid = treat as decline.
- **Timing:** Prompt appears after victory screen `input("Press Enter to continue.")` clears. New console.print then prompt.
- **Live context:** Rich Live is exited before this prompt (Phase 6 established pattern — Live cannot be active during `input()` calls).

### Narrow Terminal Detection

- **Detection point:** At render call time, not at module import time.
- **Threshold:** `console.width < 40` — strict less-than.
- **Fallback chain:** Full layout → narrow layout (no art). No intermediate breakpoints for MVP.
- **tmux/SSH/VS Code:** No special handling — width detection via `console.width` covers these automatically.

> Source: D-08, UI-06, Phase 10 success criteria item 5.
